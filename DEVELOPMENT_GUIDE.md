# Admin 설계정책서 개발 지침

이 프로젝트의 메뉴별 설계는 개발 착수 시 기준 문서로 사용합니다. 개발 요청, 이슈, PR, 작업 브랜치에서는 메뉴명을 길게 풀어 쓰기보다 화면에 표시되는 개발 참조 코드를 함께 적습니다.

## 지시 방식

권장 지시문:

```text
ADM-MEM-001 설계대로 회원 기본조회 화면을 개발해.
ADM-WAL-004 설계 기준으로 출금 승인/보류 화면의 목록, 상세, 권한, 감사로그를 구현해.
ADM-RISK-002 정책에서 벗어나는 부분은 변경 사유를 남기고 대체 정책을 제안해.
```

메뉴 상세 화면의 `지침 복사` 버튼을 누르면 해당 메뉴의 표준 개발 지시문을 복사할 수 있습니다.

## 코드 체계

메뉴 코드는 상단 업무 도메인과 CSV 순서를 기준으로 자동 부여합니다.

| 상단 메뉴 | Prefix | 예시 |
| --- | --- | --- |
| 대시보드 | `ADM-DASH` | `ADM-DASH-001` |
| 회원·고객확인 | `ADM-MEM` | `ADM-MEM-001` |
| 거래·마켓 관리 | `ADM-MKT` | `ADM-MKT-001` |
| 입출금·지갑 | `ADM-WAL` | `ADM-WAL-001` |
| 준법·FDS 관리 | `ADM-RISK` | `ADM-RISK-001` |
| 거래지원·상품 | `ADM-LIST` | `ADM-LIST-001` |
| 정산·회계 | `ADM-SETTLE` | `ADM-SETTLE-001` |
| 고객지원·서비스 | `ADM-CS` | `ADM-CS-001` |
| 보안·관리자통제 | `ADM-SEC` | `ADM-SEC-001` |

## 설계 기준 범위

개발자가 메뉴 코드를 참조하면 아래 항목을 함께 따라야 합니다.

- 화면 목적
- 업무 설명
- 탭 구성 또는 탭 제안
- 설계 방식 제안
- 주요 액션
- 권한과 감사로그 기준
- 정책 체크리스트
- 구 어드민 매핑이 있는 경우 기존 업무 흐름

## 구현 원칙

설계서와 구현이 다를 때는 코드에 임의로 맞추지 말고 아래 내용을 작업 기록에 남깁니다.

- 대상 메뉴 코드
- 달라지는 설계 항목
- 변경 사유
- 대체 정책
- 영향 범위
- 승인 또는 확인이 필요한 담당자

## 코드 변경 규칙

CSV의 상단 메뉴나 행 순서를 바꾸면 메뉴 코드가 달라질 수 있습니다. 이미 개발이 시작된 메뉴는 코드 변경 영향을 확인한 뒤 진행합니다.

운영 중인 개발 이슈나 PR에서 참조 중인 코드는 임의로 재배치하지 않습니다. 메뉴를 추가해야 한다면 같은 도메인 마지막에 추가하는 방식을 우선합니다.

## ADM-RISK-003 차단 사유코드 관리

`ADM-RISK-003`은 단순 공통코드 화면이 아니라 입출금 차단 보고 기준을 관리하는 정책 화면입니다.

개발 시 반드시 아래 기준을 함께 구현합니다.

- 차단 사유 번호 1~13 관리
- 코인거래소용 `가상자산 입출고 관련` 사유 관리
- 원화거래소용 `예치금 입출금 관련` 사유 관리
- 사유별 건 수 산출 기준 관리
- 산출 기준 값: `사건발생일`, `가상자산종목수`, `이용자수`
- 입금, 출금, 입고, 출고 차단 방향 관리
- 사유코드 변경 이력, 사용 여부, 승인 상태, 감사로그 관리

`ADM-RISK-003 설계대로 개발`이라는 요청은 위 기준표와 산출 기준을 포함해 구현하라는 의미입니다.

## ADM-RISK-001/002 차단 등록과 해제

`ADM-RISK-001`은 차단을 등록하는 화면이고, `ADM-RISK-002`는 이미 차단된 대상자의 차단을 해제하는 화면입니다. 두 화면은 단순 등록/해제 버튼이 아니라 승인 기반 고위험 업무로 구현합니다.

`ADM-RISK-001 차단 등록` 개발 기준:

- 고객 대상자를 지정한 뒤 차단 등록
- ADM-RISK-003 차단 사유코드 선택
- 로그인, 거래, 입금, 출금, 입고, 출고 등 차단 범위 선택
- FDS, KYT, KYC 미흡, 제재국가, 트래블룰 미준수 등 자동 등록 이벤트 연계
- 수동 등록과 자동 등록 구분
- 등록 요청, 승인, 반려, 적용 결과, 전후 상태 감사로그 기록

`ADM-RISK-002 차단 해제` 개발 기준:

- FDS에 걸린 고객, 수동 차단 등록 대상자, KYC·제재·트래블룰 사유 제한 고객의 해제 요청 처리
- 원 차단 사유, 등록자, 승인자, 차단 범위, 관련 Case 확인
- 해제 사유, 증빙, 조치 결과 입력
- 전체 해제와 부분 해제 구분
- 승인 완료 후 해제 적용
- 해제 요청, 승인, 반려, 보류, 적용 결과 감사로그 기록

## ADM-RISK-004 차단 관리대장

`ADM-RISK-004`는 차단 사유별 일별·월별 집계와 최초 보고 근거를 관리하는 화면입니다. 원천 로그를 단순 합산하는 화면이 아니라 `ADM-RISK-003`의 산출 기준에 따라 숫자, 날짜, 전체, 종목수, 금액을 다르게 표시해야 합니다.

개발 기준:

- 차단 등록, 자동 차단 로그, 사건 기록형 등록, 수동 보정을 원천으로 관리
- 일별 집계와 월별 집계 제공
- `이용자수` 기준은 회원 중복 제거 후 숫자 합계
- `가상자산종목수` 기준은 종목 수 또는 종목 목록, 전체 대상이면 `전체`
- `사건발생일` 기준은 날짜 목록 또는 발생일 수
- 가상자산 입고/출고/입출고 차단과 예치금 입금/출금/입출금 차단을 구분
- 가상자산 출고 차단 대상 금액과 출금 차단 대상 예치금 금액을 별도 집계
- 최초 보고일자, 최초보고 문서번호, 차단사유 근거 관리
- 수동 보정은 보정 전후 값, 보정 사유, 승인자, 증빙을 기록

정기점검, 전산장애, 메인넷 업그레이드, 하드포크, 토큰스왑 같은 사건은 등록했다고 고객 차단을 자동 생성하면 안 됩니다. 이 경우 사건 발생일과 보고값만 관리대장 집계에 반영합니다.
